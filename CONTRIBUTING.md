# 贡献指南

感谢你有兴趣为 har-to-skill 项目做贡献！

## 📋 行为准则

### 我们的承诺
为了营造一个开放和友好的环境，我们贡献者和维护者承诺让每个人都能无骚扰地参与我们的项目和社区。

### 我们的标准
有助于建立积极环境的行为包括：
- 使用友好和包容的语言
- 尊重不同的观点和经验
- 优雅地接受建设性批评
- 关注对社区最有利的事情
- 对其他社区成员表现出同理心

## 🚀 如何贡献

### 报告问题
1. 在提交新Issue之前，请先搜索现有Issue，避免重复
2. 使用清晰的标题和描述
3. 提供尽可能多的复现步骤
4. 包含你使用的版本和环境信息

### 提交功能请求
1. 清晰描述功能的使用场景和价值
2. 尽可能提供实现思路或示例

### 提交代码
1. Fork 本仓库
2. 为你的功能或修复创建一个新分支：`git checkout -b feature/my-new-feature` 或 `git checkout -b fix/my-bug-fix`
3. 提交你的更改：`git commit -am 'Add some feature'`
4. 推送到分支：`git push origin feature/my-new-feature`
5. 创建一个 Pull Request

## 🛠️ 开发规范

### 代码风格
- 遵循PEP 8规范
- 使用4空格缩进
- 函数和变量使用下划线命名法
- 类名使用大驼峰命名法
- 代码必须包含必要的注释和文档字符串

### 测试要求
- 所有新功能必须包含对应的单元测试
- 修复bug必须添加回归测试
- 提交代码前确保所有测试通过：`pytest tests/`
- 尽量保持测试覆盖率不降低

### 提交信息规范
提交信息应遵循以下格式：
```
<类型>: <主题>

<正文>
```

类型包括：
- `feat`: 新功能
- `fix`: 修复bug
- `docs`: 文档更新
- `style`: 代码格式调整（不影响代码运行的变动）
- `refactor`: 重构（既不是新增功能，也不是修改bug的代码变动）
- `perf`: 性能优化
- `test`: 测试相关
- `chore`: 构建过程或辅助工具的变动

示例：
```
feat: 添加对GraphQL API的支持
fix: 修复路径参数识别错误的问题
docs: 更新安装和使用文档
```

## ✅ Pull Request 流程

1. PR标题应清晰描述变更内容
2. PR描述应包含：
   - 变更的目的和解决的问题
   - 主要的变更点
   - 关联的Issue（如果有）
3. 所有CI检查必须通过
4. 至少需要一个维护者的批准才能合并
5. 合并前请将分支rebase到最新的main分支

## 📄 许可证
当你提交代码到本项目时，即表示你同意你的代码将在MIT许可证下发布。
